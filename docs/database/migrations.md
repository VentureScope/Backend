# Database Migrations Guide

This guide covers how to work with Alembic database migrations in the VentureScope Backend.

## Overview

We use [Alembic](https://alembic.sqlalchemy.org/) for database schema version control. Alembic automatically tracks database schema changes and allows you to upgrade/downgrade your database structure safely.

## Quick Reference

### Common Commands

```bash
# Apply all pending migrations
alembic upgrade head

# Create a new migration (auto-detect changes from models)
alembic revision --autogenerate -m "Description of changes"

# Check current migration version
alembic current

# Show migration history
alembic history

# Downgrade by 1 migration
alembic downgrade -1

# Downgrade to specific version
alembic downgrade 93e091e52124

# Downgrade to base (remove all migrations)
alembic downgrade base
```

### Environment Variables

Set `DATABASE_URL` before running any migration commands:

```bash
# For local development
export DATABASE_URL=postgresql+asyncpg://venturescope:venturescope@localhost:5432/venturescope

# For testing
export TEST_DATABASE_URL=postgresql+asyncpg://venturescope:venturescope@localhost:5432/venturescope_test
```

## How to Create a New Migration

### 1. Modify Your Models

Make changes to your SQLAlchemy models in `app/models/`:

```python
# app/models/user.py
class User(Base):
    __tablename__ = "users"
    
    id: Mapped[str] = mapped_column(String(36), primary_key=True)
    email: Mapped[str] = mapped_column(String(255), unique=True)
    # Add new field
    phone_number: Mapped[str | None] = mapped_column(String(20), nullable=True)
```

### 2. Generate Migration

```bash
# Auto-generate migration from model changes
alembic revision --autogenerate -m "Add phone_number to users table"
```

This creates a new file in `alembic/versions/` like `a1b2c3d4e5f6_add_phone_number_to_users_table.py`.

### 3. Review Generated Migration

Always review the generated migration file before applying it:

```python
def upgrade() -> None:
    # ### commands auto generated by Alembic - please adjust! ###
    op.add_column('users', sa.Column('phone_number', sa.String(length=20), nullable=True))
    # ### end Alembic commands ###

def downgrade() -> None:
    # ### commands auto generated by Alembic - please adjust! ###
    op.drop_column('users', 'phone_number')
    # ### end Alembic commands ###
```

### 4. Apply Migration

```bash
alembic upgrade head
```

## Development Workflow

### Local Development

1. **Start database services:**
   ```bash
   docker compose up -d postgres redis
   ```

   ```bash
   docker compose up backend
   # Migrations run automatically before the API starts
   ```


## Migration Best Practices

### 1. Always Review Auto-Generated Migrations

Alembic's autogenerate is helpful but not perfect:

- **Review before committing** - Check that the generated migration matches your intent
- **Test both directions** - Verify upgrade and downgrade work correctly
- **Check for data loss** - Be careful with operations that might delete data

### 2. Backward Compatibility

When possible, make changes that are backward compatible:

```python
# Good: Add nullable column
op.add_column('users', sa.Column('phone_number', sa.String(20), nullable=True))

# Careful: Add non-nullable column (requires default or data migration)
op.add_column('users', sa.Column('phone_number', sa.String(20), nullable=False, server_default=''))
```

### 3. Data Migrations

For complex data transformations, write custom migration code:

```python
def upgrade() -> None:
    # Schema change
    op.add_column('users', sa.Column('full_name', sa.String(255)))
    
    # Data migration
    connection = op.get_bind()
    connection.execute(text("""
        UPDATE users 
        SET full_name = CONCAT(first_name, ' ', last_name)
        WHERE first_name IS NOT NULL AND last_name IS NOT NULL
    """))
    
    # Remove old columns
    op.drop_column('users', 'first_name')
    op.drop_column('users', 'last_name')
```

### 4. Testing Migrations

Always test your migrations:

```bash
# Test upgrade
alembic upgrade head

# Test downgrade
alembic downgrade -1

# Test upgrade again
alembic upgrade head
```

## Troubleshooting

### Common Issues

#### 1. "Target database is not up to date"

```bash
# Check current version
alembic current

# Check history
alembic history

# Upgrade to latest
alembic upgrade head
```

#### 2. "Can't locate revision identified by 'head'"

This usually means there are no migrations yet:

```bash
# Create initial migration
alembic revision --autogenerate -m "Initial migration"
```

#### 3. "FAILED: Can't locate revision identified by '...'"

The migration file might be missing or corrupted:

```bash
# Check if migration files exist
ls alembic/versions/

# Check alembic_version table
psql -h localhost -U venturescope -d venturescope -c "SELECT * FROM alembic_version;"
```

#### 4. "Multiple head revisions are present"

You have multiple migration branches. Merge them:

```bash
# Show branches
alembic branches

# Merge branches
alembic merge -m "Merge migration branches" head1 head2
```

#### 5. Database Connection Issues

Check your DATABASE_URL:

```bash
# Verify database is running
docker compose ps postgres

# Test connection
psql $DATABASE_URL -c "SELECT 1;"
```

### Recovery Procedures

#### Reset to Clean State (Development Only)

```bash
# WARNING: This deletes all data!

# Drop all tables
alembic downgrade base

# Recreate from latest migration
alembic upgrade head
```

#### Rollback Last Migration

```bash
# Downgrade one step
alembic downgrade -1

# Fix the issue, then upgrade again
alembic upgrade head
```

#### Manual Version Stamping

If the alembic_version table gets out of sync:

```bash
# Stamp database with specific version (without running migration)
alembic stamp 93e091e52124

# Or stamp as current head
alembic stamp head
```

## File Structure

```
alembic/
├── versions/           # Migration files (tracked in git)
│   └── 93e091e52124_initial_migration_user_table.py
├── env.py             # Alembic environment configuration
├── script.py.mako     # Template for new migration files
└── README             # Alembic readme

alembic.ini            # Alembic configuration file
scripts/
└── run_migrations.sh  # Helper script for running migrations
```

## Integration with Docker

Migrations are automatically run when starting the backend service:

```yaml
# docker-compose.yml
backend:
  command: >
    sh -c "
      ./scripts/run_migrations.sh &&
      uvicorn app.main:app --host 0.0.0.0 --port 8000 --reload
    "
```

This ensures the database is always up-to-date when the API starts.
